# Celerity — Stellar Setup & Dependencies

Read this before Phase 0 of the build plan. It covers exactly what to install, what to generate, and — importantly — what you do NOT need (no paid API keys, no anchor accounts).

## TL;DR on the two things you asked
- **Does this involve anchors?** Yes conceptually (they're your cash-out and your "why Stellar"), but for the hackathon the anchor is a **stub** — a mock function that shows USD/stablecoin becoming a PHP number. You do **not** integrate a real anchor, because that needs a licensed VASP partner and compliance onboarding (weeks of partnership work, not code). No judge expects it.
- **What API keys do you need?** Essentially **none**. Everything runs on Stellar Testnet, which is free: accounts are funded by a faucet (Friendbot), the RPC endpoint is public, and the only "keys" are Stellar keypairs you generate yourself with the CLI. Do not sign up for anchor/exchange APIs — it's a time sink with no demo payoff.

## What to install (versions current as of mid-2026)

### 1. Rust (v1.84.0 or higher — required for the Stellar wasm target)
macOS / Linux:
```
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
# restart terminal, then verify:
rustc --version    # must be >= 1.84.0
```
Windows: download and run `rustup-init.exe`, accept defaults, restart terminal.

### 2. The WebAssembly target
Stellar contracts compile to Wasm. Recent toolchains use `wasm32v1-none`; some tutorials still show `wasm32-unknown-unknown`. Install the one the current docs specify:
```
rustup target add wasm32v1-none
# (if a command later complains it wants the older target, also run:)
# rustup target add wasm32-unknown-unknown
```
Note: the wasm target is per-toolchain — if you update Rust, reinstall it.

### 3. A C build system (needed to build the CLI from source, and generally useful)
Debian / Ubuntu:
```
sudo apt update && sudo apt install -y build-essential
```
macOS: `xcode-select --install`

### 4. The Stellar CLI (this is your Swiss-army knife — build, deploy, invoke)
```
cargo install stellar-cli --features opt
stellar --version
```
(The command is `stellar`; older docs call it `soroban` — same tool, newer name.)

### 5. Node.js (v18+ — for the oracle signer and the frontend)
Install from nodejs.org or via nvm. Verify:
```
node --version   # >= 18
```

### 6. VS Code + rust-analyzer extension (recommended editor)

## Stellar setup (all free, all Testnet)

### Configure the Testnet network
```
stellar network add --global testnet \
  --rpc-url https://soroban-testnet.stellar.org:443 \
  --network-passphrase "Test SDF Network ; September 2015"
```

### Generate + fund your dev account (Friendbot funds it automatically)
```
stellar keys generate --global alice --network testnet
stellar keys address alice
```
This creates a keypair AND funds it with test XLM via Friendbot (the Testnet faucet). No signup, no card, no API key. Make a second identity (e.g. `funder2`) later for the multi-funder demo.

### Initialize the project (Phase 0)
```
stellar contract init celerity
cd celerity
```
This scaffolds a Rust workspace with the recommended structure. Replace the sample contract with the Celerity contract from the doc's §10.

### Build, deploy, invoke (the loop you'll repeat all week)
```
# build
cargo build --target wasm32v1-none --release

# deploy to Testnet -> returns a CONTRACT_ID (save it)
stellar contract deploy \
  --wasm target/wasm32v1-none/release/celerity.wasm \
  --source-account alice \
  --network testnet \
  --alias celerity

# invoke a function
stellar contract invoke \
  --id <CONTRACT_ID> \
  --source-account alice \
  --network testnet \
  -- <function_name> --arg value
```

## The keys in this project (and how to handle them safely)

| Key | What it is | Where it lives | Committed? |
|---|---|---|---|
| `alice` (dev account) | Deploys + calls the contract | Stellar CLI keystore | Never |
| `funder2`, etc. | Extra funders for multi-funder demo | Stellar CLI keystore | Never |
| Oracle signing key | Ed25519 key the Node signer uses to sign weather events | `.env`, generated locally | **Never — .gitignore it** |
| Contract ID | Address of your deployed contract | Config/`.env`, or a constant | Fine to commit (it's public) |

Rules:
- The oracle key is generated by your Node signer script and read from `.env`. Add `.env` to `.gitignore` on the first commit.
- Never paste any secret key into a chat, a screenshot, or committed code.
- Testnet keys hold only worthless test XLM, but treat them correctly anyway — it builds the right habit for Mainnet.

## Node dependencies (oracle signer + frontend)

Oracle signer (Ed25519 signing — matches what the contract verifies):
```
npm install @stellar/stellar-sdk dotenv
```
(`@stellar/stellar-sdk` gives you keypair generation and Ed25519 signing; `dotenv` loads the key from `.env`.)

Frontend (React + Stellar SDK):
```
npm create vite@latest celerity-web -- --template react
cd celerity-web
npm install @stellar/stellar-sdk
```
The frontend talks to the same public Testnet RPC — no key needed for reads; user actions are signed by the connected account.

## What you explicitly do NOT need (don't waste time here)
- ❌ Any anchor / Coins.ph / Maya / exchange API key or account.
- ❌ Any paid RPC provider — the public Testnet RPC is fine for a demo.
- ❌ KYC / VASP onboarding of any kind.
- ❌ Real USDC/PHPC — use test assets or a simple mock token on Testnet.
- ❌ Mainnet anything. Everything is Testnet until long after the hackathon.

## The anchor, honestly (for your pitch)
In the demo, the anchor step is a labeled mock: your frontend takes the released on-chain amount and displays "→ ₱X via Stellar anchor (Coins.ph-class)." On stage you say: *"Everything on-chain is real and live; the fiat cash-out is where a licensed anchor plugs in — I mock that leg and I'm explicit about it."* That honesty is a strength, not a gap. The real SEP-31 integration is roadmap, gated by a licensing partnership, not by your code.
